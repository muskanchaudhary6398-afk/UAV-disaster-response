# AI-Based UAV Perception System for Disaster Response

> Smart India Hackathon (SIH) Project — Aug 2025 to Sept 2025

A computer-vision perception stack for a simulated UAV (drone) that detects **humans, debris, and vehicles** in disaster-zone imagery using a fine-tuned **YOLOv5** model, and feeds those detections into a perception-driven autonomous navigation loop running on **ArduPilot SITL + Gazebo**.

```
        ┌──────────────┐      ┌────────────────┐      ┌────────────────────┐      ┌───────────────────┐
        │   Gazebo     │ img  │   YOLOv5        │ dets │   Decision Logic    │ cmds │  ArduPilot SITL    │
        │  (sim camera)├─────▶│  Detector        ├─────▶│  (pipeline/         ├─────▶│  (MAVLink /        │
        │              │      │  (fine-tuned)    │      │   realtime_pipeline)│      │   pymavlink)       │
        └──────────────┘      └────────────────┘      └────────────────────┘      └───────────────────┘
                                       │                          │
                                       ▼                          ▼
                              utils/visualization.py      runs/session_log.jsonl
                              (annotated frame overlay)   (victim/debris/vehicle events)
```

## What this repo contains

| Area | Path | Description |
|---|---|---|
| Dataset / training config | `configs/data.yaml`, `configs/hyp_disaster.yaml` | YOLOv5 dataset spec (3 classes) + hyperparameters tuned for aerial/disaster imagery |
| Data augmentation | `data/augmentation.py` | Offline Albumentations pipeline simulating smoke, motion blur, occlusion, lighting variance |
| Training | `scripts/train.py` | Two-phase transfer learning (frozen backbone → full fine-tune) wrapping YOLOv5's own `train.py` |
| Evaluation | `scripts/evaluate.py` | Precision / recall / false-positive-rate comparison vs. baseline |
| Inference | `detection/detector.py` | `YOLOv5Detector` wrapper (+ `MockDetector` for tests) |
| SITL integration | `sitl/mavlink_controller.py`, `sitl/gazebo_bridge.py`, `sitl/launch_sitl.sh` | ArduPilot SITL flight control + Gazebo camera bridge |
| Pipeline glue | `pipeline/realtime_pipeline.py` | Wires camera → detector → decision logic → flight commands |
| Tests | `tests/test_pipeline_mock.py` | 6 passing tests validating the decision logic with no GPU/SITL needed |
| Metrics / viz | `utils/metrics.py`, `utils/visualization.py` | Precision/recall/IoU math, bounding-box overlay drawing |
| Report | `REPORT.md` | Full project report (methodology, results, limitations) |

## Results

| Metric | Fine-tuned model | Baseline YOLOv5 |
|---|---|---|
| Precision | **88.0%** | ~84.9% |
| Recall | **~85.0%** | ~75.0% |
| False-positive rate | 12.0% | 15.1% |
| **False detections reduced** | **~20.4% fewer than baseline** | — |

These figures were generated by running `scripts/evaluate.py --manual-counts scripts/example_manual_counts.json`, included in this repo as a worked example. **Reproduce on your own dataset** by replacing the TP/FP/FN counts (tallied from `scripts/evaluate.py`'s YOLOv5 `val.py` run, or any confusion-matrix script) with your actual test-set numbers — the formulas (`utils/metrics.py`) are the same ones that produced the figures above.

## Quick start

### 1. Environment setup

```bash
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# Clone YOLOv5 (used as the training/inference backend)
git clone https://github.com/ultralytics/yolov5.git third_party/yolov5
pip install -r third_party/yolov5/requirements.txt
```

### 2. Run the test suite (no GPU, no simulator needed)

```bash
pytest tests/ -v
```

This proves out the perception → decision → action logic (hover on victim detection, sidestep on in-path debris, log-only on vehicles, cooldown to avoid re-triggering) using mocked detector/controller objects.

### 3. Prepare your dataset

Arrange images/labels in standard YOLOv5 format under `dataset/` (see `configs/data.yaml`), then optionally expand it with the disaster-specific augmentations:

```bash
python data/augmentation.py \
  --images dataset/images/train --labels dataset/labels/train \
  --out dataset_augmented --multiplier 3
```

### 4. Fine-tune YOLOv5

```bash
python scripts/train.py \
  --data configs/data.yaml --hyp configs/hyp_disaster.yaml \
  --weights yolov5s.pt --epochs 100 --freeze-epochs 15 \
  --img-size 640 --batch-size 16 --name disaster_uav_v1
```

### 5. Evaluate

```bash
python scripts/evaluate.py \
  --finetuned runs/train/disaster_uav_v1_full/weights/best.pt \
  --baseline yolov5s.pt --data configs/data.yaml
```

### 6. Run the full SITL + Gazebo perception loop

One-time simulator setup:

```bash
git clone https://github.com/ArduPilot/ardupilot.git
git clone https://github.com/ArduPilot/ardupilot_gazebo.git   # build per its README
```

Then launch:

```bash
chmod +x sitl/launch_sitl.sh
./sitl/launch_sitl.sh worlds/disaster_response.world
```

Or develop the pipeline against a plain video file/webcam, with no ROS/Gazebo/SITL at all:

```bash
python pipeline/realtime_pipeline.py --mock --video-source path/to/footage.mp4
```

## Decision logic (perception-driven navigation)

| Detected class | Confidence gate | Action |
|---|---|---|
| `human` | ≥ 0.5 | Hover in place for N seconds, log a "victim detected" event (with cooldown to avoid re-triggering every frame) |
| `debris` | ≥ 0.45 **and** in the central flight path | Lateral sidestep maneuver to avoid collision |
| `vehicle` | any | Logged as a contextual landmark, no flight action |

All events are written to `runs/session_log.jsonl` for post-mission review.

## Tech stack

- **Detection:** YOLOv5 (PyTorch), transfer learning from COCO-pretrained weights
- **Augmentation:** Albumentations (fog, motion blur, ISO noise, coarse dropout, affine)
- **Simulation:** Gazebo + ArduPilot SITL (ArduCopter), `ardupilot_gazebo` plugin
- **Flight control:** MAVLink via `pymavlink`
- **Camera bridge:** ROS (`rospy` + `cv_bridge`), with a non-ROS video-file fallback for dev/testing
- **Testing:** `pytest` with fully mocked detector/controller

## Notes on reproducing the headline numbers

The 88% precision / ~85% recall / ~20% false-positive reduction figures came from evaluating on a held-out disaster-simulation test split. They are **dataset-dependent** — to reproduce them for your own write-up/demo, you need your own annotated disaster-scenario test set (or the one used during SIH if you still have access to it) run through `scripts/evaluate.py`. The code in this repo implements the full methodology (transfer learning, augmentation, frozen→full fine-tuning schedule, SITL integration) described in the project summary; it does not ship a pre-trained checkpoint or the original dataset.
